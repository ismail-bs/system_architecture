Project Highlights

Architecture / System Design

Layered architecture: core → domain → infrastructure → utils → middlewares → app.js

Clear separation of concerns

Fully OOP-based: services, entities, repositories, command objects

Dependency Injection container to manage services and repositories

Design Patterns Included

Singleton → Database connection (core/database.js)

Factory → Notification creation (infrastructure/notifications/notificationFactory.js)

Observer → Event bus for domain events (infrastructure/events/eventBus.js)

Decorator → Logging wrapper for service methods (infrastructure/decorators/loggerDecorator.js)

Strategy → Notification strategies: Email/SMS/Push (notificationFactory.js)

Proxy → Caching repository reads (infrastructure/cache/proxyCache.js)

Adapter → Legacy email system adapter (infrastructure/notifications/legacyEmailAdapter.js)

Command → Commands for user registration & task creation (infrastructure/commands/commandHandler.js)

Builder → Step-by-step creation of users & tasks (utils/builder/*.js)

Middleware → Request validation & logging (middlewares/validator.js & logger.js)

Repository → Abstraction of DB operations (infrastructure/repositories/*.js)

Dependency Injection → Container registering & injecting services and repositories (core/container.js)

SOLID Principles

S → Single Responsibility: Each class/file handles one responsibility

O → Open/Closed: Adding new notifications or commands does not modify existing code

L → Liskov Substitution: All strategies follow same interface

I → Interface Segregation: Small, focused interfaces in domain & infrastructure

D → Dependency Inversion: High-level modules depend on abstractions, not concrete classes

Other Features

Express endpoints:

POST /users → register user

GET /users → list users

POST /tasks → create task

GET /tasks → list tasks

Fully commented to explain every pattern, SOLID principle, and OOP design decision

Ready-to-run: npm install → npm start

Minimal in-memory database for simplicity (Singleton)
